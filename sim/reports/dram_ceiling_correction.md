# DRAM 利用率上限补充分析

> 前一份报告假设 DRAM 利用率可以逼近 100%，这个假设不成立。

---

## DRAM 效率的实际天花板

### 已知损耗（不可消除）

| 损耗来源 | 占比 | 性质 |
|---------|:---:|------|
| **Refresh** (tRFC/tREFI) | 5.4% | 物理定律，JEDEC 强制 |
| **Row conflict** (随机访问) | ~4.5% | 权重 tile 跨行时触发 precharge+activate |
| **Bus turnaround** (读→写切换) | 1-2% | DDR 半双工，权重只读但激活写入 SRAM 后可能回写 |
| **Burst 碎片** | 1-3% | 8 KB tile 不是 burst_size 整数倍时末 burst 浪费 |

**理论天花板 ≈ 85-88%。** 这就是 config 里 `dram_efficiency: 0.85` 的来源。

### 报告中的错误

报告中"DRAM 利用率 78%"用的是**已打折的 43.5 GB/s** 做分母：

```
DRAM 需求 = 33.8 GB/s（3B 模型权重数据）
有效带宽 = 51.2 × 0.85 = 43.5 GB/s
利用率 = 33.8 / 43.5 = 78%
```

这意味着**相对物理极限（85% 天花板）而言，实际可争取的余量比我说的 22% 小得多：**

```
物理天花板:   85%
当前利用率:   78% (相对天花板 78/85 = 92%)
──────────────────────────
可争取余量:    7%  不是 22%
```

---

## 修正后的瓶颈视图

```
有效带宽 43.5 GB/s 的分配（3B @19.9 tok/s）：

┌──────────────────────────────────────────┬──────┐
│██████████████████████████████████████████│      │
│         权重搬运 33.8 GB/s (78%)          │ 余量 │
│                                          │ 9.7  │
└──────────────────────────────────────────┴──────┘

距离物理天花板仅剩 7% 余量。
1.5B 模型 (39.0 GB/s = 90%) 已经贴墙。
```

| 模型 | 有效利用率 | 占天花板% | 实际余量 |
|------|:---:|:---:|:---:|
| 1.5B | 90% | **106% → 已超** ⚠️ | 0%（已在交换） |
| 3B | 78% | 92% | **7%** |
| 7B | 76% | 89% | 9% |

> 1.5B 在 43.3 tok/s 时 DRAM 需求 39.0 GB/s，已超出 85% 天花板。这意味着 **1.5B 的实际 tok/s 已经被 DRAM 限制了** — 可能 simulator 对 DMA/Compute 重叠建模偏乐观。

---

## 这意味着什么

修正后排名不变，但紧迫性变了：

| 排名 | 瓶颈 | 紧迫度 |
|:--:|---|:--:|
| **#1** | Pipeline (385c/tile) | 主瓶颈，优化空间大 |
| **#2** | DRAM 带宽 | **几乎同步到墙**，Pipeline 优化稍大一点就会撞 |

### 关键结论

**Pipeline 和 DRAM 是联合瓶颈，不是先后关系。**

优化 pipeline → tok/s 从 19.9 往 25+ 走 → DRAM 需求从 78% 往 95%+ 飙 → 还没看到 pipeline 优化收益就被 DRAM 卡死。

所以如果要提升性能，**必须 pipeline 和 DRAM 同时动**：
- Pipeline：double-buffer tile 重叠 fill/drain，或用 Block 引擎替代 Systolic
- DRAM：加通道（64→128 bit），或换 HBM

只动一个，另一个立刻变成新墙。
