项目概述

CaduceusCore 是一颗通用 NPU 协处理器，同时面向 CV（YOLOv8/ViT/ResNet）和 LLM（Qwen 3B/7B）推理。 核心设计约束：性能瓶颈在 DRAM 带宽，不在算力。 项目遵循严格的三阶段开发流程——Arc Model（架构沙盘与设计空间搜索）→ Func Model（Golden Reference 与 RTL 规格）→ RTL（硬件实现与验证闭环）。 截至 2026-06-27，已完成 12 个 Plans，总计约 107 AI-assisted hours，全部 Plans 状态均为 APPROVED。

关键发现

  • S2 和 S3 共用同一颗 die（block 80×1536, 124mm²）—— 两颗芯片覆盖三个产品线
  • LPDDR5 场景 BW 瓶颈：所有引擎 TPS 集中在 21–23，选 FSA 面积最优
  • On-chip 3D DRAM BW 随面积线性增长（7.5 GB/s/mm²），大芯片 = 多 TOPS + 多 BW
  • seq_len 对 TTFT 影响巨大：1024 需要 ≥72 TOPS，128 只需 4 TOPS
  • SRAM 在 on-chip 场景零敏感（512KB 足够），在 LPDDR5 场景有意义（4MB 最优）
  • 面积模型经 TPUv1 ISCA 2017 die-shot 数据校准，不再偏高

Bootstrap 配置说明

Func Model 目前只细化了其中一个起点——64×64 Broadcast MAC 阵列 + SFU + Vector Engine。 这是项目的 bootstrap 配置，用于建立 RTL 开发流程、验证工具链和 bit-exact 对比方法论。 它不是 Arc DSE 推荐架构（FSA 或 block）的直接实现，而是先跑通全流程的最小可行硬件。 后续 Func Model 会按 DSE 结论切换到推荐引擎（FSA / block），届时重新生成 golden reference。

关键教训

LL-001

性能模型不等于精度验证

INT4 全局 scale 方案在 Func Model 首次跑真实 GGUF 权重时发现完全不可用（rel_err 10³–10⁴）。 Layer 1 时序模型只把 weight_precision_bits=4 当作字节数计算的除数，从未用真实 Transformer 权重做 INT4 量化→矩阵乘→对比验证。 Layer 2 软件协议走 float32 反量化路线，绕过了硬件 INT4 通路。时序假设被默认当成了「已验证的精度方案」。

在搭建任何时序模型之前，先建数值模型验证精度方案能不能跑通。时序模型只回答「多快」，不回答「对不对」。

修复： 1. Func Model (golden_executor.py) 承担数值模型角色；2. 新增验证门禁：新量化方案必须通过 all-layer PASS；3. Func Model 输出作为 $readmemh golden reference。
