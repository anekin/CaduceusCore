## 〇、市场与产品定义（MRD）

### 0.1 市场时机

边缘 AI 市场 2024 年估值 ~$220 亿，预计 2026 年达 ~$476 亿，年复合增长率约 30%（Fortune Business Insights）。驱动力：

- **3B 以下模型质量跃升**：MiniCPM 4.0（0.5B）超过 Qwen3-0.6B；Gemma 3 270M 在 Synaptics Coral NPU 上跑通；INT4 量化让 3B 模型仅占 1.5 GB
- **LLM 推理从云端向边缘迁移**："The On-Device LLM Revolution"（Edge AI Vision, 2026/04）——延迟、隐私、成本三重推力
- **Agent 爆发**：2025 年 35 位大模型创业者一致看好智能体，端侧 Agent 需要本地推理能力
- **数据合规与隐私**：金融、医疗、政务场景强制数据不出域

### 0.2 缺口：3B LLM 推理的硬件真空

| 层级 | 现有方案 | 能跑 3B LLM？ | 痛点 |
|------|---------|:---:|------|
| **MCU 级 NPU** | Ethos-U55, Kendryte K230 | ❌ | <1 TOPS，只跑视觉/关键词 |
| **边缘视觉 NPU** | Hailo-8 (26 TOPS), Edge TPU (4 TOPS) | ❌ | 架构为 CNN 设计，编译器只认视觉模型，不支持 Transformer decode |
| **手机 SoC NPU** | Apple ANE, 高通 Hexagon (45 TOPS) | ⚠️ 勉强 | 集成在 SoC 内不可独立部署；功耗和散热约束极严 |
| **数据中心卡** | NVIDIA A100/H100, FuriosaAI 2nm | ✅ | 功耗 150-700W，价格 $8K-$30K，不是边缘产品 |
| **PCIe 边缘卡** | Lanner Falcon H8 (Hailo-8×N) | ❌ | 仍是视觉 NPU 堆叠，Transformer 支持差 |
| **CPU 软跑** | x86/ARM 跑 llama.cpp | ⚠️ 慢 | 3B @ INT4 在 M5 上 30 tok/s，在 RPi 5 上 ~2 tok/s |

**结论**：20-25 tok/s、15W、$100-200 价格区间的 **3B LLM 专用 PCIe 推理卡——不存在**。这是产品定位。

### 0.4.1 国内竞对分析

| 厂商 | 芯片 | 形态 | INT8算力 | 3B LLM可用？ | 定位 | 与我们关系 |
|------|------|------|------|:---:|------|------|
| **算能** | BM1684X | PCIe卡/M.2模组 | 32 TOPS | ⚠️ 宣称支持但未见实测 | 边缘推理卡，最直接竞品 | ⚠️ 直接竞争 |
| **华为昇腾** | Ascend 310B | PCIe卡/模组 | 22 TOPS | ❌ 面向CV推理 | 全栈自研，生态封闭 | 不同赛道 |
| **瑞芯微** | RK3588 | SoC集成 | 6 TOPS | ❌ 算力差5-8× | SoC，不是独立加速卡 | 互补 |
| **寒武纪** | 思元220 | 模组 | 8 TOPS | ❌ 边缘CV推理 | 云边端全覆盖，市值高但边缘出货少 | 不同量级 |
| **地平线** | 征程5/6 | SoC集成 | 128 TOPS | ❌ 车规ADAS专用 | 汽车芯片，不是通用推理 | 无关 |
| **爱芯元智** | AX630/AX650 | SoC集成 | ~10 TOPS | ❌ 视觉SoC | 安防/机器人ISP+NPU | 无关 |

**核心结论**：

1. **算能 BM1684X 是唯一直接竞品**——32 TOPS，PCIe/M.2 形态，TPU 架构，宣称支持大模型但市场验证不足。但他们没有 Transformer 原生 SFU，也没有 MLIR/IREE 开放软件栈（用的是自研 TpuMLIR）
2. **其余全是 SoC 集成或专用场景**——瑞芯微/地平线/爱芯的 NPU 是搭在 CPU+ISP 旁边的，不是独立加速卡
3. **国内没有「M.2 模组 + 3B LLM 专用」的产品**——大家都在说"支持大模型"但实际是 CV NPU 改的，和我们有本质差

### 0.5 竞争优势

1. **LLM+CV 双栈原生支持**：systolic array 统一跑 GEMM，SFU 覆盖 Transformer 和 CNN 全套算子。Hailo/Coral 只能跑 CV
2. **开放软件栈**：MLIR/IREE 开源编译器，不像 Hailo 闭源工具链锁死模型格式。客户可以部署自训模型
3. **PCIe 标准形态**：即插即用，不像手机 NPU 集成在 SoC 里拿不出来
4. **成本结构**：12nm 成熟工艺 + ~30mm² 小 die + FCSP 封装，BOM 对比 Tegra/Orin 有绝对优势
5. **产业链兼容**：M.2 2230 M-key 是 Hailo-8 和 Coral 已经验证的标准形态，设备厂商不需要改 PCB、不需要新供应链、不需要新装配流程。同一个槽，换一张卡，从"只能跑 CV"升级到"CV+LLM 全吃"

### 0.5.1 产品形态策略

同一颗 30mm² NPU 芯片，三种产品形态覆盖不同场景：

```
            ┌──────────────────────┐
            │   NPU 芯片 (同一颗)    │
            │   30mm² / $7 / 10-12W │
            └──────────┬───────────┘
                       │
       ┌───────────────┼───────────────┐
       │               │               │
  M.2 2230 模组      PCIe 卡        USB 算力棒
  (主力形态)       (高功率场景)     (便携场景)
  $79 OEM          $129 MSRP       $99 MSRP
  22×30mm          HHHL            60×20mm
```

| | M.2 2230 模组 | PCIe 卡 | USB 算力棒 |
|------|:---:|:---:|:---:|
| **目标客户** | 设备厂商贴板集成 | 工业/服务器/个人台式 | 笔记本/平板/开发者 |
| **部署方式** | 厂家预装，终端用户无感 | 用户自插 | 即插即用 |
| **量级** | 万-百万 | 百-千 | 千-万 |
| **散热** | 主机散热系统 | 独立被动散热片 | 棒状铝壳 |
| **对主控要求** | 需 M.2 M-key 槽 | 需 PCIe x4 槽 | 需 USB4/Type-C |
| **厂商接受度** | ✅ 现有供应链直接兼容 | ✅ 标准 PCIe | ✅ 但 USB 棒形态新 |

**M.2 2230 为什么是主力**：

设备厂商（工控机、智能音箱、翻译机、NVR）已经在用 M.2 2230 槽装 Hailo-8 和 Coral。用同一个物理形态、同一个电气接口、同一个供电方案——他们不需要重新设计主板，只需把 Hailo-8 模组换成我们的模组，从"只能跑 CV"升级到"LLM+CV 双栈"。这种零改动的升级路径是设备厂最容易接受的采纳方式。

### 0.6 风险与劣势

1. **量产依赖 IP 授权**：PCIe PHY 和 LPDDR5 PHY 需采购商用 IP，NRE ~$1M 计入研发投入，不摊入单颗成本
2. **软件栈建设周期**：MLIR HAL 后端从零开发，12-16 周工作量大
3. **客户生态冷启动**：不像 NVIDIA 有 CUDA 社区——第一批客户需要深度支持
4. **单点性能天花板**：64-bit LPDDR5 锁死 25 tok/s，3B 以上模型力不从心
5. **市场教育成本**："你的 NPU 才 33 TOPS？Hailo 有 26 啊"——需要让客户理解 TOPS ≠ LLM 性能。但 CV 也能跑这个卖点有助于打破"只管 LLM"的刻板印象

### 2.2 计算阵列

| 参数 | 取值 | 依据 |
|------|------|------|
| 阵列规模 | 128×128 MAC | prefill 2000 token 耗时 0.37s |
| 时钟频率 | 1 GHz | TSMC 12nm 可达到 |
| MAC 位宽 | INT4×INT8→INT32 | 权重 INT4，激活 INT8 |
| 峰值算力 | ~33 TOPS (INT8) | 128²×2×1GHz，INT8 双倍 |
| 数据流 | Weight-Stationary | 权重静态，减少数据搬移功耗 |
| 阵列数量 | 1 个 | 放在 15W 功耗预算内 |

### 2.3 内存子系统

| 层级 | 规格 | 容量 | 带宽 | 用途 |
|------|------|------|------|------|
| L1 SRAM | 片上 scratchpad | 256 KB×2 | TB/s 级 | 阵列输入/输出缓存 |
| L2 SRAM | 共享 SRAM | 2 MB | ~500 GB/s | 中间激活、KV 缓存头 |
| L3 DRAM | LPDDR5-6400 64-bit | 4-8 GB | 51.2 GB/s | 模型权重（1.5GB）+ KV 缓存 + I/O |

**带宽验证**：
- Decode 20 tok/s：1.5 GB × 20 = 30 GB/s → LPDDR5 51.2 GB/s ✅（1.7x 裕量）
- Decode 25 tok/s：含 KV Cache + 激活约 2.0 GB/token → 50 GB/s，刚好满足
- Decode 30 tok/s：60 GB/s，64-bit 不够 \[→ 升级 128-bit 可达\]

### 2.4 特殊函数单元（SFU）

Transformer 和 CNN 特有的非矩阵运算，无法在 systolic array 上高效执行，需独立硬件。Systolic array 本身是统一的 GEMM 引擎——CNN 卷积通过 im2col 变换为矩阵乘法后同一阵列处理。

| 算子 | 硬件方案 | 数据格式 | 适用模型 |
|------|---------|---------|---------|
| **Softmax** | 分段指数 + 减法归一化，流水线化 | BF16 | LLM + CV 分类头 |
| **Layer Normalization** | 均值/方差并行计算 + 融合乘加 | BF16 | LLM |
| **GELU 激活** | 查表 + 线性插值近似 | BF16 | LLM |
| **RoPE 编码** | CORDIC 旋转单元 | BF16 | LLM |
| **Max/Avg Pooling** | 滑动窗口比较器 | BF16/INT8 | YOLO, ResNet |
| **ReLU** | 组合逻辑（取 0 或 x） | BF16/INT8 | 大部分 CNN |
| **SiLU/Swish** | 可配（复用 GELU 的查表逻辑） | BF16 | YOLOv8+ |

参考：中国科学院 2024 年发表的统一 Softmax+LayerNorm 硬件架构，可作为即插即用单元集成。Pooling 和 ReLU 为轻量级逻辑（共几千门电路），面积增量 < 1mm²。

### 2.5 KV Cache 管理

| 参数 | 取值 | 说明 |
|------|------|------|
| Cache 存储位置 | LPDDR5 | 容量大（4-8 GB），足够长上下文 |
| Cache 格式 | INT8 | 省带宽 |
| 2000 token 开销 | 128 MB（估算） | 3B×2×2层×2000×2B/KV ≈ 96 MB |
| 地址管理 | 硬件页表 | RISC-V 核维护，DMA 搬移 |

无需片上 SRAM 存全部 KV Cache。L2 SRAM 缓存最近的 token 即可。

## 四、软件栈

### 4.1 编译器工具链

```
PyTorch / ONNX
      │
      ▼
Torch-MLIR / StableHLO  ───→ MLIR 方言层
      │
      ▼
IREE Compiler ───→ Linalg / Vector 方言
      │
      ▼
HAL (Hardware Abstraction Layer) ← 自定义 NPU 后端
      │
      ▼
NPU ISA ───→ 指令序列 → 板级 Driver
```

### 4.2 各层职责

| 层 | 职责 | 参考 |
|------|------|------|
| Torch-MLIR | PyTorch 模型 → MLIR | 上游开源 |
| IREE Flow | 图优化、算子融合 | 上游开源 |
| IREE HAL | 硬件抽象，自定义后端接口 | **需开发** |
| NPU ISA | 控制 systolic array、SFU、DMA | **需设计** |
| Driver | PCIe 通信、命令队列 | **需开发** |

**关键复用**：Google Coral NPU 的 IREE/MLIR 工具链已在 Synaptics Torq 上验证，架构范式可直接套用。主要工作量在 HAL 后端和 NPU ISA 设计。

### 4.3 运行时

| 组件 | 说明 |
|------|------|
| Host Runtime | 模型加载、tokenizer、推理调度（跑在 x86 主控上） |
| NPU Firmware | 指令解码、DMA 调度、中断处理（跑在 RISC-V 核上） |

## 五、指令集架构（NPU ISA）概要

| 指令类别 | 示例 | 说明 |
|---------|------|------|
| 矩阵运算 | `MMUL srcA, srcB, dst, M, N, K` | 触发 systolic array |
| 特殊函数 | `SOFTMAX src, dst, len` / `LAYERNORM ...` | SFU 调用 |
| 数据搬移 | `DMA_LOAD dram_addr, sram_addr, size` | DRAM↔SRAM |
| 同步 | `BARRIER` / `WAIT_MMUL` | 流水线同步 |
| 控制 | `LOOP count, pc_offset` / `JMP` | 程序流 |

设计思想：CISC 风格粗粒度指令，一条指令对应一个完整的算子调用。RISC-V 核负责任务级调度，不参与逐 MAC 控制。

## 六、开源基础复用矩阵

| 模块 | 复用源 | 复用方式 |
|------|--------|---------|
| Systolic Array 架构 | OpenTPU / TPUv1 | 参考 256×256→缩放至 128×128 |
| 数据流设计方法论 | Gemmini | 参数空间探索（尺寸/数据流/数据类型） |
| RISC-V 核集成 | Coral NPU | 参考 RISC-V 与加速器总线的连接范式 |
| MLIR/IREE 工具链 | Coral NPU + IREE 上游 | 自定义 HAL 后端 |
| PCIe 控制器 IP | 商用 IP | 采购（Synopsys/Cadence） |
| LPDDR5 控制器 IP | 商用 IP | 采购 |
| Softmax/LayerNorm 硬件 | 学术论文 | 中科院统一架构 |

**Fork 策略**：不从任何单一项目 fork。以 OpenTPU 的 systolic array 结构为参考，用 Verilog 写定制化 128×128 版本，集成商用 IP（PCIe/DRAM 控制器），软件栈 fork Coral NPU 的 MLIR 流程。
